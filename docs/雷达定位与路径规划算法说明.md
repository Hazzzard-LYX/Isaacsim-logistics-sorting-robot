# 雷达定位与路径规划算法说明

本文档说明当前项目中“雷达定位”和“路径规划”的实际实现方式、算法原理、数据流以及当前局限。

对应代码文件：

- `run_path_planning_demo.py`：完整巡检流程、路径规划、机器人运动控制。
- `lidar_localization.py`：独立的 2D LiDAR 已知地图定位原型。
- `create_logistics_field.py`：场地、障碍物、机器人和传感器挂载点生成。

## 1. 当前系统整体关系

当前项目中，路径规划和雷达定位分成两个层次：

1. `run_path_planning_demo.py` 中的完整巡检演示已经可以驱动机器人移动。
2. `lidar_localization.py` 中实现了一个独立的 LiDAR 定位算法原型。
3. 但完整巡检流程目前还没有真正接入 `lidar_localization.py` 的 scan matching 结果。

也就是说，当前完整流程中路径规划使用的是仿真场景提供的“真值位姿”，不是雷达算法估计出来的位置。

当前完整流程中的位姿来源是：

```text
/World/LogisticsField/Robot/Sensors/Lidar 的世界坐标
        ↓
get_lidar_pose()
        ↓
Pose2D(x, y, yaw)
        ↓
A* 路径规划
```

这里的 LiDAR prim 主要作为“传感器安装点”和“位姿读取点”，还不是实际的激光测距数据源。

## 2. 当前完整流程中的雷达位姿

### 2.1 雷达模型

场景中机器人顶部创建了一个简化 2D LiDAR，路径为：

```text
/World/LogisticsField/Robot/Sensors/Lidar
```

该 prim 在 `create_logistics_field.py` 中创建，并写入了一些传感器属性，例如：

```text
sensor:type = planar_lidar
sensor:minRange = 0.05
sensor:maxRange = 6.0
sensor:horizontalFovDeg = 360.0
sensor:horizontalResolution = 720
```

这些属性描述了一个平面激光雷达的预期参数，但当前完整巡检流程没有从 Isaac Sim 中读取真实 LiDAR ranges。

### 2.2 当前 get_lidar_pose() 的作用

完整流程中的位姿读取函数在 `run_path_planning_demo.py`：

```python
def get_lidar_pose(stage: Usd.Stage, robot_yaw: float) -> Pose2D:
    lidar = stage.GetPrimAtPath(LIDAR_PATH)
    world_tf = UsdGeom.XformCache().GetLocalToWorldTransform(lidar)
    p = world_tf.ExtractTranslation()
    return Pose2D(float(p[0]), float(p[1]), robot_yaw)
```

它做了三件事：

1. 找到场景中的 LiDAR prim。
2. 读取 LiDAR prim 的世界坐标。
3. 把该坐标作为机器人当前 `(x, y)`，并使用控制器内部维护的 `robot_yaw` 作为朝向。

因此，当前完整巡检中的“雷达定位”本质上是：

```text
直接读取仿真真值坐标，而不是根据激光扫描数据估计位姿。
```

这种做法适合课程设计早期验证，因为它能先保证路径规划、机器人移动、目标巡检和识别反馈流程跑通。

## 3. 独立雷达定位原型

虽然完整巡检尚未接入真实雷达定位，但项目中已经有一个独立的 LiDAR 定位算法原型：`lidar_localization.py`。

该文件不依赖 Isaac Sim，可以单独运行：

```bash
python lidar_localization.py
```

### 3.1 已知地图建模

`KnownFieldMap` 表示已知二维地图。地图由三类几何组成：

1. 场地边界：4m x 2m 的矩形边框。
2. 箱体：取货区和放置区的矩形轮廓。
3. 障碍物：两个圆柱障碍物，在二维中表示为圆。

代码中对应的数据结构包括：

```python
@dataclass(frozen=True)
class Segment:
    x1: float
    y1: float
    x2: float
    y2: float

@dataclass(frozen=True)
class Circle:
    x: float
    y: float
    radius: float
```

矩形边界和箱体会被拆成线段，圆柱障碍物则保留为圆。

### 3.2 Ray Casting 模拟 LiDAR

`ray_cast()` 用于模拟一束激光从机器人位置发射出去后，首先撞到哪个物体。

输入：

```text
机器人位置: (x, y)
激光方向: angle
最大量程: max_range
```

输出：

```text
该方向上最近命中的距离
```

它会分别计算：

1. 射线与所有线段的交点距离。
2. 射线与所有圆的交点距离。
3. 取所有有效交点中的最小距离。
4. 如果没有命中，则返回最大量程。

核心逻辑：

```text
for 每条地图线段:
    计算射线和线段是否相交
    如果相交，记录距离

for 每个圆形障碍物:
    计算射线和圆是否相交
    如果相交，记录距离

返回最近命中距离
```

### 3.3 Scan Matching 定位

雷达定位的核心思想是 scan matching：

```text
如果机器人处在某个候选位姿 (x, y, yaw)，
根据已知地图可以预测一帧 LiDAR 扫描距离。

把预测扫描和实际观测扫描进行比较，
误差最小的候选位姿就是估计位姿。
```

在 `KnownMapLocalizer.localize()` 中，输入包括：

```python
ranges         # 观测到的一帧雷达距离
initial_guess  # 初始位姿猜测
```

算法从初始猜测附近搜索候选位姿，并逐层缩小搜索范围。

搜索层级：

```python
search_levels = [
    (0.30, math.radians(20.0), 0.05, math.radians(5.0)),
    (0.08, math.radians(6.0), 0.02, math.radians(1.5)),
    (0.025, math.radians(2.0), 0.005, math.radians(0.5)),
]
```

每一层包含：

```text
xy_radius: x/y 搜索半径
yaw_radius: yaw 搜索半径
xy_step: x/y 搜索步长
yaw_step: yaw 搜索步长
```

也就是说，算法先粗略搜索，再在更小范围内精细搜索。

### 3.4 误差函数

`scan_error()` 用于评价一个候选位姿好不好。

步骤：

1. 根据候选位姿生成预测 scan。
2. 将预测距离和观测距离逐束比较。
3. 计算平方误差。
4. 对单束误差做上限裁剪，避免异常点影响过大。
5. 返回平均误差。

核心形式：

```text
error = mean(min((observed_range - predicted_range)^2, 0.25))
```

候选位姿误差越小，说明它越符合当前 LiDAR 观测。

### 3.5 当前测试结果

`lidar_localization.py` 中构造了一个理想测试：

```python
true_pose = Pose2D(0.35, -0.22, math.radians(18.0))
initial_guess = Pose2D(0.10, -0.05, math.radians(5.0))
```

脚本先用真值位姿生成一帧模拟 LiDAR scan，然后从有误差的初始猜测开始定位。

运行结果：

```text
true_pose:
  x=0.350, y=-0.220, yaw=18.00 deg
initial_guess:
  x=0.100, y=-0.050, yaw=5.00 deg
estimated_pose:
  x=0.350, y=-0.220, yaw=18.00 deg
```

这说明在理想已知地图和无噪声模拟数据下，scan matching 可以正确恢复机器人位姿。

## 4. 路径规划算法

完整流程中的路径规划实现在 `run_path_planning_demo.py`，核心函数是：

```python
astar_path(start_xy, goal_xy)
```

当前路径规划不是写死整条轨迹，而是在机器人每次前往下一个目标点前，重新计算当前段路径。

### 4.1 固定任务顺序

巡检目标由 `build_inspection_targets()` 生成。

当前目标顺序是固定的：

```text
pickup_1
pickup_2
pickup_3
place_4
place_5
place_6
place_7
place_8
返回初始位置
```

因此，任务点顺序是固定的。

但每两个目标点之间的具体移动路径不是预先写死的，而是由 A* 算法根据当前起点和目标点计算。

### 4.2 地图离散化

A* 在二维网格上搜索路径。

网格分辨率：

```python
GRID_RESOLUTION = 0.05
```

即每个网格边长为 0.05m，也就是 5cm。

连续坐标会被转换成整数网格：

```python
def grid_key(point):
    return (
        round(point[0] / GRID_RESOLUTION),
        round(point[1] / GRID_RESOLUTION),
    )
```

整数网格也可以转换回连续坐标：

```python
def grid_xy(key):
    return (key[0] * GRID_RESOLUTION, key[1] * GRID_RESOLUTION)
```

### 4.3 可通行区域判断

一个网格点是否可走，由 `is_free()` 判断：

```python
def is_free(x: float, y: float) -> bool:
    return in_field(x, y) and not occupied_by_obstacle(x, y)
```

它包含两个条件：

1. 点必须在场地边界内。
2. 点不能落入障碍物膨胀区域。

### 4.4 场地边界约束

`in_field()` 会给场地边缘留出安全距离：

```python
BOUNDARY_MARGIN = 0.18
```

场地尺寸为：

```python
FIELD_LENGTH = 4.0
FIELD_WIDTH = 2.0
```

机器人中心允许活动范围约为：

```text
x: -2.0 + 0.18 到 2.0 - 0.18
y: -1.0 + 0.18 到 1.0 - 0.18
```

即：

```text
x: -1.82 到 1.82
y: -0.82 到 0.82
```

这样机器人不会贴着边界走。

### 4.5 障碍物膨胀

场地中间有两个圆柱障碍物：

```python
OBSTACLE_POSITIONS = [
    ("obstacle_1", (-0.8, 0.0)),
    ("obstacle_2", (0.8, 0.0)),
]
```

圆柱物理半径：

```python
OBSTACLE_RADIUS = 0.051
```

路径规划中还额外加入安全膨胀：

```python
OBSTACLE_CLEARANCE = 0.28
```

因此，A* 判断障碍物占据时使用：

```text
障碍物占据半径 = OBSTACLE_RADIUS + OBSTACLE_CLEARANCE
              = 0.051 + 0.28
              = 0.331m
```

如果某个网格点距离任一圆柱中心小于等于 0.331m，则认为该点不可通行。

这样做的目的：

1. 考虑机器人本体尺寸。
2. 留出安全余量。
3. 避免规划路径擦着障碍物边缘通过。

### 4.6 起点/终点修正

如果起点或目标点刚好落在不可通行区域，`nearest_free_key()` 会向周围搜索最近的可通行网格。

流程：

```text
先检查原始网格是否可通行
如果不可通行:
    从半径 1 开始向外扩展搜索
    找到所有可通行候选点
    选择距离原始点最近的网格
```

这可以避免目标点太靠近障碍物或边界时 A* 直接失败。

### 4.7 A* 搜索过程

`astar_path()` 使用标准 A* 搜索。

主要数据结构：

```python
open_heap      # 优先队列，存放待扩展节点
came_from      # 记录每个节点从哪个节点来，用于最后回溯路径
cost_so_far    # 从起点走到当前节点的真实代价
```

每次从 `open_heap` 中取出优先级最低的节点进行扩展。

优先级计算：

```text
priority = 已走代价 + 启发代价
```

其中：

```text
已走代价 = 从起点到当前节点的路径长度
启发代价 = 当前节点到目标节点的欧氏距离
```

### 4.8 邻域扩展

当前 A* 使用 8 邻域：

```text
左、右、上、下、左上、右上、左下、右下
```

对应代码：

```python
for dx in (-1, 0, 1):
    for dy in (-1, 0, 1):
        if dx == 0 and dy == 0:
            continue
```

直线移动代价为：

```text
0.05m
```

斜向移动代价为：

```text
sqrt(2) * 0.05m
```

因此路径可以沿斜线绕开障碍物，不局限于横平竖直移动。

### 4.9 路径回溯

当搜索到目标网格后，算法根据 `came_from` 从终点反向回溯到起点：

```python
path_keys = [goal]
while path_keys[-1] != start:
    path_keys.append(came_from[path_keys[-1]])
path_keys.reverse()
```

最后把网格坐标转回世界坐标：

```python
return [grid_xy(k) for k in path_keys]
```

得到的结果是一串二维路径点：

```text
[(x0, y0), (x1, y1), ..., (xn, yn)]
```

### 4.10 路径简化

A* 输出的原始路径包含大量相邻网格点。为了减少机器人运动抖动，项目使用 `simplify_path()` 保留方向变化点。

例如原始路径：

```text
A -> B -> C -> D -> E
```

如果 A 到 E 基本是同一方向，中间 B/C/D 就会被省略。

简化后路径更像：

```text
起点 -> 转弯点1 -> 转弯点2 -> 终点
```

这样机器人执行路径时不需要每 5cm 都调整一次目标点。

## 5. 路径执行方式

路径执行由 `move_along_path()` 完成。

当前机器人不是通过真实轮速、电机或物理控制移动，而是直接修改 `/World/LogisticsField/Robot` 根节点位姿。

### 5.1 平移速度

当前平移速度：

```python
LINEAR_SPEED = 0.35
```

单位是 m/s。

### 5.2 角速度限制

当前转向角速度上限：

```python
ANGULAR_SPEED = 0.75
```

单位是 rad/s，约等于：

```text
0.75 rad/s ≈ 43 deg/s
```

转向通过 `step_yaw_towards()` 实现，用来避免机器人朝向瞬间跳变。

### 5.3 控制周期

控制更新周期：

```python
UPDATE_DT = 1.0 / 60.0
```

即 60Hz。

每一帧根据当前点到下一个路径点的方向，计算这一帧应该移动的距离：

```text
step = min(LINEAR_SPEED * UPDATE_DT, dist)
```

然后更新机器人位置。

### 5.4 全向底盘假设

代码注释中说明当前假设机器人是麦轮/全向底盘。

因此机器人可以沿路径方向平移，同时车身朝向平滑跟随路径方向。

平移方向使用的是：

```text
当前位置 -> 目标路径点
```

而不是受限后的车身 yaw。

这样可以避免因为车身转向滞后导致机器人走弧线或绕着路径点打转。

## 6. 在线规划流程

完整巡检过程中，每到一个新目标前都会重新规划。

主流程如下：

```text
读取当前位姿
    ↓
选择下一个巡检目标
    ↓
A* 根据当前起点和目标点计算路径
    ↓
简化路径
    ↓
机器人沿路径移动
    ↓
到达目标点后转向箱体并停留
    ↓
执行模拟视觉检测
    ↓
进入下一个目标
```

对应代码结构：

```python
for target in targets:
    current_pose = get_lidar_pose(stage, current_pose.yaw)
    raw_path = astar_path((current_pose.x, current_pose.y), target.stand_xy)
    path = simplify_path(raw_path)
    current_pose = move_along_path(stage, translate_op, orient_op, current_pose, path)
    current_pose = rotate_to_face_and_wait(...)
```

因此，当前项目中“在线计算路径”的含义是：

```text
每段路径都根据当前位姿和当前目标重新计算，
而不是提前写死整条机器人运动轨迹。
```

但需要注意，地图和障碍物本身是静态已知的，没有实时更新。

## 7. 当前实现的局限

### 7.1 雷达定位局限

当前完整流程中的雷达位姿存在以下局限：

1. 没有读取真实 LiDAR ranges。
2. 没有把 `lidar_localization.py` 的 scan matching 接入巡检流程。
3. `get_lidar_pose()` 读取的是仿真真值坐标。
4. yaw 不是通过雷达估计，而是由当前控制器内部维护。

因此，当前完整流程不能说已经实现了真实雷达定位闭环。

更准确的说法是：

```text
当前系统预留了雷达定位接口，并实现了独立的已知地图 scan matching 原型；
完整巡检流程暂时使用仿真真值位姿作为定位输入。
```

### 7.2 路径规划局限

当前路径规划也有一些限制：

1. 巡检目标顺序固定，不做任务顺序优化。
2. 地图静态，不处理动态障碍物。
3. A* 障碍物检查目前主要考虑圆柱障碍物和边界。
4. 箱体虽然是场景几何，但在完整路径规划的 `is_free()` 中没有作为独立障碍物膨胀处理。
5. 机器人运动是强制位姿更新，不是真实底盘动力学控制。

## 8. 后续接入真实雷达定位的方案

后续如果要把独立雷达定位接入完整巡检流程，可以按以下数据流改造：

```text
Isaac Sim LiDAR 传感器输出 ranges
        ↓
整理为 lidar_localization.py 需要的 angles + ranges
        ↓
KnownMapLocalizer.localize(ranges, initial_guess)
        ↓
得到估计位姿 Pose2D(x, y, yaw)
        ↓
替换 run_path_planning_demo.py 中的 get_lidar_pose()
        ↓
A* 使用真实定位结果重新规划路径
```

改造重点：

1. 将场景中的 LiDAR prim 替换或扩展为 Isaac 可输出 scan 的传感器。
2. 保证 scan 的角度定义和 `make_lidar_angles()` 一致。
3. 将上一帧估计位姿作为下一帧 `initial_guess`。
4. 在 `get_lidar_pose()` 中返回 scan matching 的估计结果，而不是 USD prim 世界坐标。

完成后，路径规划才可以说是真正使用了雷达定位结果。
