# 机电一体化课程设计项目详细说明文档

## 1. 项目概述

本项目是一个基于 Isaac Sim / Omniverse USD 场景的物流比赛任务仿真系统。项目围绕一个简化物流赛场展开，完成从场地生成、真实机器人模型导入、路径巡检、取货动作、豆类分拣、卸货动作到可视化演示的完整流程。

当前版本的主目标不是做底层动力学控制，而是建立一个可展示、可调参、可解释的任务级仿真流程：

1. 自动生成物流比赛场地。
2. 导入 `final-urdf` 中的真实机器人视觉模型。
3. 规划机器人从初始位置到取货区、放置区、回到起点的路径。
4. 在取货区识别箱内物料颜色，并执行夹爪取料动作。
5. 根据物料类型控制 `bank` 挡板，将物料导入不同料仓。
6. 在放置区识别数字标签，并执行对应料仓的卸货动作。
7. 保留静态场景打开脚本和完整巡检脚本，便于分别检查模型/场地和完整流程。

当前完整流程入口是：

```bash
cd /home/hazzzard/机电一体化课程设计
env ISAAC_HEADLESS=0 PYTHONUNBUFFERED=1 /home/hazzzard/anaconda3/envs/env_isaaclab/bin/python run_path_planning_demo.py
```

只打开静态场景的入口是：

```bash
cd /home/hazzzard/机电一体化课程设计
env ISAAC_HEADLESS=0 PYTHONUNBUFFERED=1 /home/hazzzard/anaconda3/envs/env_isaaclab/bin/python open_logistics_field.py
```

两者的区别非常重要：`open_logistics_field.py` 只生成并打开场景；`run_path_planning_demo.py` 会执行完整巡检、取货、分拣、卸货流程。

## 2. 项目目录和关键文件

当前项目目录下文件较多，其中很多是调参过程中的备份文件。当前主流程主要依赖以下文件：

此外，项目中有大量形如 `*_backup.py` 的文件。这些是每次较大调整前保留的历史版本，主要用于回退和对比。当前运行不依赖这些备份文件。

## 3. 场地生成逻辑

场地由 `create_logistics_field.py` 生成，输出到 `logistics_field_stage.usd`。核心入口是：

```python
def main(output_usd: str = OUTPUT_USD) -> str:
    stage = create_new_stage(output_usd)
    build_logistics_field(stage)
    saved = save_stage(stage, output_usd)
    return saved
```

### 3.1 坐标系约定

项目使用标准 USD/Isaac Sim 坐标：

| 轴 | 含义 |
|---|---|
| `x` | 场地长度方向 |
| `y` | 场地宽度方向 |
| `z` | 竖直向上 |

场地中心位于世界原点 `(0, 0, 0)`，主场地底面贴在世界地面上。

### 3.2 场地缩放

当前场地和场地内物体使用统一缩放：

```python
FIELD_SCALE = 2.0
```

这意味着原始 4 m x 2 m 场地被放大为：

```text
FIELD_LENGTH = 8.0 m
FIELD_WIDTH  = 4.0 m
```

放大的是场地、货箱、围栏、障碍物、地面标识等场地元素；机器人真实模型保持原尺寸。这样做是为了让真实机器人模型与赛场比例更协调，避免模型相对场地过大。

### 3.3 场地元素

`build_logistics_field()` 负责组装完整场景，主要包括：

1. 世界底板 `WorldFloor`。
2. 主场地地板 `Floor`。
3. 四周红色围栏 `Fences`。
4. 中心线、起始区、中心区域和取货/放置参考线 `GroundMarks`。
5. 取货区 `PickupZone`。
6. 放置区 `PlaceZone`。
7. 中部圆柱障碍物 `Obstacles`。
8. 机器人根节点和传感器挂载点 `Robot/Sensors`。
9. 场景灯光和默认观察相机。

### 3.4 取货区

取货区由三个货位组成：

```python
PICKUP_SLOTS = [
    ("pickup_1", (-1.75 * FIELD_SCALE, 0.6 * FIELD_SCALE), "vertical"),
    ("pickup_2", (-1.75 * FIELD_SCALE, 0.0), "vertical"),
    ("pickup_3", (-1.75 * FIELD_SCALE, -0.6 * FIELD_SCALE), "vertical"),
]
```

每个取货位包含：

1. A4 纸堆叠置物台。
2. 无盖透明货箱。
3. 箱内液体/物料填充块。
4. `vision:liquidColor` 自定义属性，用于当前仿真中的颜色识别。

当前颜色为：

```python
LIQUID_COLORS = {
    "yellow": (0.95, 0.78, 0.08),
    "green": (0.05, 0.75, 0.15),
    "white": (0.95, 0.95, 0.90),
}
```

对应任务语义为黄豆、绿豆和白芸豆。当前代码中已经不再使用红色液体作为取货物料。

### 3.5 放置区

放置区由五个货箱组成：

```python
PLACE_SLOTS = [
    ("place_4", (1.76 * FIELD_SCALE, 0.8 * FIELD_SCALE), 4, "horizontal"),
    ("place_5", (1.8 * FIELD_SCALE, 0.4 * FIELD_SCALE), 5, "vertical"),
    ("place_6", (1.8 * FIELD_SCALE, 0.0), 6, "vertical"),
    ("place_7", (1.8 * FIELD_SCALE, -0.4 * FIELD_SCALE), 7, "vertical"),
    ("place_8", (1.76 * FIELD_SCALE, -0.8 * FIELD_SCALE), 8, "horizontal"),
]
```

每个放置箱会生成一个外侧数字标签，标签数字来自：

```python
DIGIT_LABEL_NUMBERS = [1, 2, 3, 4, 5]
```

数字标签每次运行会随机打乱，除非设置固定 `RANDOM_SEED`。标签 prim 上会写入：

```text
logistics:labelNumber
```

当前主流程通过读取这个属性模拟视觉识别数字。

## 4. 真实机器人模型导入

真实机器人模型由 `final_robot_model.py` 导入。当前使用的模型路径是：

```python
SOURCE_URDF = WORKSPACE / "final-urdf" / "urdf" / "final-urdf.urdf"
SOURCE_MESH_DIR = WORKSPACE / "final-urdf" / "meshes"
```

导入后的机器人视觉模型路径为：

```text
/World/LogisticsField/Robot/FinalModel
```

机器人根节点仍然是：

```text
/World/LogisticsField/Robot
```

主流程移动的是 `/World/LogisticsField/Robot` 根节点，因此真实模型作为子树会跟随整体移动。

### 4.1 URDF 运行期转换

Isaac Sim 的 URDF 导入器对 `package://` mesh 路径要求较严格。`prepare_runtime_urdf()` 会把 `final-urdf` 中的 STL 复制到 `.final_urdf_runtime/meshes`，并把 URDF 中的相对 mesh 路径替换为绝对路径：

```text
final-urdf/urdf/final-urdf.urdf
        ↓
.final_urdf_runtime/final_robot.urdf
```

这样可以避免不同包名、中文包名或路径解析导致导入失败。

### 4.2 初始位姿修正

真实模型导入后会应用固定偏移：

```python
FINAL_MODEL_REAR_SHIFT = 0.4
FINAL_MODEL_OFFSET = Gf.Vec3d(0.1465 - FINAL_MODEL_REAR_SHIFT, -0.25, 0.0)
FINAL_MODEL_YAW_OFFSET = -math.radians(12.0)
FINAL_MODEL_GROUND_Z = 0.008
```

这些值是通过多轮可视化调试得到的，目的是让真实机器人模型在视觉上与原有方块机器人版本的初始位姿保持一致。

### 4.3 颜色和初始夹爪姿态

当前 `final_robot_model.py` 会将以下分拣/卸货相关部件设置为红色：

```python
RED_SORTING_PARTS = ("l_bank", "r_bank", "l_gate", "r_gate", "f_gate")
```

静态场景打开时会额外调用：

```python
import_final_robot_model(stage, apply_home_pose=True)
```

这会把夹爪设置为：

```text
claw-roll = -1.57
claw = open
```

完整巡检脚本中也会在启动后设置同样的初始夹爪姿态。

## 5. 完整巡检流程

完整任务由 `run_path_planning_demo.py` 执行。主入口是：

```python
def main() -> None:
    saved_path = build_scene(OUTPUT_USD)
    ctx.open_stage(saved_path)
    import_final_robot_model(stage)
    execute_inspection_route(stage)
```

整体流程如下：

```text
生成场地 USD
        ↓
打开 stage
        ↓
导入真实机器人模型
        ↓
读取初始位姿
        ↓
构建巡检目标
        ↓
依次前往 3 个取货箱
        ↓
识别物料颜色并执行取货动作
        ↓
依次前往 5 个放置箱
        ↓
识别数字标签并执行对应卸货动作
        ↓
返回初始位置
        ↓
保持 Isaac Sim 窗口打开
```

### 5.1 目标点生成

目标点由 `build_inspection_targets()` 生成。取货区和放置区分别使用不同的停靠距离：

```python
PICKUP_BOX_VIEW_DISTANCE = 0.80
PLACE_BOX_VIEW_DISTANCE = 0.70
```

取货区还包含横向微调：

```python
PICKUP_STAND_LATERAL_OFFSET = -0.10
PICKUP_STAND_EXTRA_LATERAL_OFFSETS = {
    "pickup_1": -0.05,
}
PICKUP_STAND_EXTRA_VIEW_DISTANCES = {
    "pickup_1": -0.10,
}
```

这些参数用于避免真实模型与货箱穿模，同时保证夹爪能进入取货箱。

## 6. 路径规划与运动控制

### 6.1 当前位姿来源

主流程中的 `get_lidar_pose()` 读取的是 LiDAR prim 的世界坐标：

```python
world_tf = UsdGeom.XformCache().GetLocalToWorldTransform(lidar)
p = world_tf.ExtractTranslation()
return Pose2D(float(p[0]), float(p[1]), robot_yaw)
```

因此，当前完整流程里的“雷达定位”是仿真真值读取，不是实际 LiDAR scan matching。真正的 LiDAR 定位算法原型在 `lidar_localization.py` 中，但还没有接入完整巡检流程。

### 6.2 A* 网格规划

路径规划采用二维 A*：

```python
GRID_RESOLUTION = 0.05
BOUNDARY_MARGIN = 0.18
OBSTACLE_CLEARANCE = 0.28
```

可通行性由 `is_free()` 判断：

```text
点在场地边界内
并且不落入障碍物膨胀区
```

障碍物主要是两个圆柱障碍物，规划时按半径加安全余量膨胀：

```python
if math.hypot(x - ox, y - oy) <= OBSTACLE_RADIUS + OBSTACLE_CLEARANCE:
    return True
```

### 6.3 路径后处理

A* 网格路径会产生很多短折线。当前代码进行了两步后处理：

1. `simplify_path()`：保留方向变化点，去掉重复方向上的中间点。
2. `shortcut_path()`：如果两点之间有直线可视通路，则直接合并为一段长直线。

这样可以减少斜线行走时的方向抖动，使机器人尽量沿平滑直线运动。

### 6.4 机器人运动方式

机器人运动通过直接设置 `/World/LogisticsField/Robot` 根节点位姿实现：

```python
translate_op.Set(Gf.Vec3d(pose.x, pose.y, 0.0))
orient_op.Set(yaw_to_quat(pose.yaw))
```

平移速度和角速度为：

```python
LINEAR_SPEED = 0.35
ANGULAR_SPEED = 0.75
UPDATE_DT = 1.0 / 60.0
```

当前运动控制是任务级可视化控制，不是轮子动力学闭环控制。脚本中保留了麦轮/全向底盘的运动假设：平移方向跟随路径点，车身 yaw 平滑跟随路径方向。

## 7. 视觉识别实现边界

当前主流程中的视觉识别是仿真属性读取，不是真正从相机图像中识别。

### 7.1 取货区颜色识别

取货区颜色读取函数是：

```python
read_pickup_liquid_color(stage, pickup_name)
```

它遍历 `PickupZone` 下对应液体 prim，并读取：

```text
vision:liquidColor
```

读取结果为：

```text
yellow / green / white
```

主流程再根据 `LIQUID_TO_LAMP` 设置视觉指示灯颜色，并根据 `BANK_SORT_ANGLES` 设置分拣挡板角度。

### 7.2 放置区数字识别

放置区数字读取函数是：

```python
read_nearest_place_label_number(stage, box_center_xy)
```

它遍历 `PlaceZone` 下的标签 prim，读取：

```text
logistics:labelNumber
```

并选取离当前目标箱中心最近的标签数字。该数字用于决定是否卸货以及打开哪个料仓门。

### 7.3 后续真实视觉方案

项目已有 `深度相机视觉识别方案说明.md`，描述了后续如何用 RGB-D 图像替代当前属性读取。真正接入后，应将：

```python
read_pickup_liquid_color()
read_nearest_place_label_number()
```

替换为基于相机帧的识别函数。主流程其他部分可以基本保持不变，因为动作逻辑只依赖识别结果。

## 8. 取货机构动作

取货、分拣、卸货动作都由 `ClawAnimator` 控制。由于 URDF 导入后的 USD link 层级不一定正确传播父子变换，当前代码不依赖 USD 父子运动自动传递，而是显式重算每个关键 link 的位置和姿态。

核心受控部件包括：

| 部件 | 作用 |
|---|---|
| `f_b` / `f_b_platform` | 前后滑台。 |
| `l_r` / `l_r_platform` | 左右滑台。 |
| `claw_roll` | 夹爪翻转关节。 |
| `claw_lift` | 夹爪升降关节。 |
| `left_claw` / `right_claw` | 左右夹爪开合。 |
| `store` | 料仓主体。 |
| `l_gate` / `r_gate` / `f_gate` | 左、右、前料仓门。 |
| `l_bank` / `r_bank` | 分拣挡板。 |

### 8.1 关节范围和初始姿态

当前关键参数：

```python
FB_JOINT_MIN = -0.20
FB_JOINT_MAX = 0.10
LR_JOINT_MIN = -0.20
LR_JOINT_MAX = 0.10

CLAW_LIFT_UP = 0.0
CLAW_LIFT_DOWN = -0.50
CLAW_ROLL_HOME = -math.pi / 2.0
CLAW_ROLL_DUMP = 0.0

LEFT_CLAW_OPEN = 0.0
LEFT_CLAW_CLOSED = 0.02
RIGHT_CLAW_OPEN = 0.0
RIGHT_CLAW_CLOSED = -0.02
```

### 8.2 取货动作顺序

`run_pickup_sequence()` 的动作顺序为：

```text
夹爪打开并保持 claw-roll=-1.57
        ↓
根据夹爪与取货箱包围盒计算 f-b / l-r 水平对准量
        ↓
夹爪下降到箱内液体上方合理深度
        ↓
夹爪闭合
        ↓
夹爪上升
        ↓
claw-roll 旋转到 0
        ↓
夹爪张开
```

水平对准由 `compute_pickup_alignment()` 完成，它会读取取货箱和左右夹爪的包围盒，把夹爪下端尽量移动到箱体内部，避免从箱壁穿模。

下降高度由 `compute_pickup_down_lift()` 完成，它会读取取货箱底部和液体顶部位置，使夹爪伸入液体但不过深。

## 9. 分拣逻辑

取货区识别到颜色后，会调用：

```python
claw_animator.set_sorting_bank_for_liquid(liquid_color)
```

当前分拣挡板角度为：

```python
BANK_SORT_ANGLES = {
    "green": 0.0,
    "white": 0.70,
    "yellow": 1.32,
}
```

含义是：

| 识别颜色 | 当前 bank 角度 | 说明 |
|---|---:|---|
| `green` | `0.00` | 进入一侧料仓。 |
| `white` | `0.70` | 进入中间料仓。 |
| `yellow` | `1.32` | 进入另一侧料仓。 |

这里的角度是演示逻辑参数。若后续根据实际机构发现左右方向相反，只需要调整 `BANK_SORT_ANGLES` 的对应角度。

## 10. 卸货逻辑

放置区识别到数字后，只有标签 `1`、`2`、`3` 会执行卸货动作，标签 `4`、`5` 不执行卸货。

### 10.1 数字与料仓门映射

当前映射为：

```python
UNLOAD_GATE_BY_LABEL = {
    1: "r_gate",
    2: "l_gate",
    3: "f_gate",
}
```

对应关系：

| 标签数字 | 料仓 | 打开的门 |
|---:|---|---|
| `1` | 右侧料仓 | `r_gate` |
| `2` | 左侧料仓 | `l_gate` |
| `3` | 中间料仓 | `f_gate` |
| `4` | 不卸货 | 无 |
| `5` | 不卸货 | 无 |

### 10.2 卸货前车身转向

机器人在放置箱前先面向箱子识别数字，随后根据标签数字调整卸货朝向：

```python
UNLOAD_YAW_OFFSET_BY_LABEL = {
    1: math.pi / 2.0,
    2: -math.pi / 2.0,
    3: math.pi,
}
```

含义：

| 标签数字 | 卸货前额外旋转 |
|---:|---:|
| `1` | `+90°` |
| `2` | `-90°` |
| `3` | `180°` |

### 10.3 卸货前云台位移

为了让货物能够倒入对应箱子，卸货前会移动云台：

```python
PLATFORM_HOME_FB = -0.05
PLATFORM_HOME_LR = -0.05

UNLOAD_PLATFORM_POSE_BY_LABEL = {
    1: (0.10, PLATFORM_HOME_LR),
    2: (-0.20, PLATFORM_HOME_LR),
    3: (PLATFORM_HOME_FB, -0.20),
}
```

对应关系：

| 标签数字 | 卸货方向 | f-b | l-r |
|---:|---|---:|---:|
| `1` | 右侧料仓卸货 | `0.10` | `-0.05` |
| `2` | 左侧料仓卸货 | `-0.20` | `-0.05` |
| `3` | 中间料仓卸货 | `-0.05` | `-0.20` |

卸货完成后，云台会回到：

```text
f-b = -0.05
l-r = -0.05
```

### 10.4 卸货动作顺序

`run_unload_sequence()` 的动作顺序为：

```text
关闭所有料仓门
        ↓
移动云台到对应卸货位置
        ↓
料仓从 travel 高度抬升到 unload 高度
        ↓
打开对应料仓门
        ↓
保持一段时间
        ↓
关闭料仓门
        ↓
料仓降低回 travel 高度
        ↓
云台回到 (-0.05, -0.05)
```

关键参数：

```python
STORE_LIFT_TRAVEL = -0.25
```

每次离开停靠点之前还会调用 `reset_platform_for_departure()`，确保夹爪和云台回到统一的行驶姿态。

## 11. 雷达定位原型

`lidar_localization.py` 是一个独立的已知地图 + 2D LiDAR scan matching 原型。它不依赖 Isaac Sim，可以单独运行：

```bash
python3 lidar_localization.py
```

该原型包含：

1. `KnownFieldMap`：用线段和圆构建二维地图。
2. `ray_cast()`：根据候选位姿模拟激光束命中距离。
3. `simulate_scan()`：生成一帧模拟 LiDAR scan。
4. `KnownMapLocalizer.localize()`：在初始猜测附近做粗到细搜索，找出误差最小的位姿。
5. `scan_error()`：计算观测 scan 与预测 scan 的平方误差。

当前主流程没有接入该原型。要接入真实雷达定位，需要将 `run_path_planning_demo.py` 中的 `get_lidar_pose()` 替换为：

```text
读取 Isaac LiDAR ranges
        ↓
调用 KnownMapLocalizer.localize()
        ↓
输出 Pose2D(x, y, yaw)
        ↓
交给 A* 路径规划
```

## 12. 当前系统的实现边界

为了避免汇报中过度表述，需要明确以下边界：

1. 当前完整流程的路径规划是真实算法：使用 A* 网格搜索并做路径后处理。
2. 当前完整流程的机器人移动是强制设置 USD 根节点位姿，不是轮子动力学闭环控制。
3. 当前完整流程的视觉识别是读取 USD 属性，不是真实图像识别。
4. 当前完整流程的雷达位姿是读取 LiDAR prim 世界坐标，不是真实 LiDAR scan matching。
5. `lidar_localization.py` 已经实现了独立定位原型，但尚未与完整流程集成。
6. 真实机器人模型主要用于视觉展示和机构动作演示，不是基于 Articulation Controller 的物理关节控制。

这些边界并不影响课程设计展示主线。当前系统已经完整展示了物流任务流程、路径规划、机构协同、分拣和卸货逻辑。后续若要提高真实性，应优先把视觉和雷达从属性读取替换为传感器数据处理。

## 15. 后续改进方向

建议后续工作按以下顺序推进：

4. 将机器人根节点强制位姿更新替换为底盘速度控制，进一步贴近真实机器人。
5. 如果需要更真实的机构动作，可把当前视觉 link 重算方式升级为 Isaac Articulation 关节控制。
6. 增加任务状态日志，例如当前目标、识别结果、bank 角度、打开的料仓门、完成状态等，便于汇报录屏和答辩解释。

